The GLM-OCR migration to Ascend NPU via SGLang serving succeeded after 5 Phase 5 iterations. Three distinct, non-trivial repair patterns were discovered: (1) NumPy ≥2.0 breaks CANN 8.1.RC1 tbe/te import due to removed np.float_ alias — this is a systemic dependency constraint for ALL CANN 8.x migrations; (2) Phase 4 string-literal replacement ('cuda:' → 'npu:') leaves stale integer literals that encode the old prefix length in string-slice operations, causing device validation to reject valid NPU identifiers; (3) SGLang on NPU requires --disable-cuda-graph and --skip-server-warmup flags to avoid CUDA-specific runtime paths. A fourth borderline pattern (isinstance(dict) failing on HuggingFace BatchEncoding UserDict in SGLang serving_chat.py) was observed but is an SGLang-internal bug, not a generalizable migration experience.
