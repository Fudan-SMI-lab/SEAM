# Avoid CUDA dependency download by using --system-site-packages with pre-installed torch-npu

During Phase 2 (Virtual Environment Creation) of a CUDA-to-NPU migration, the initial attempt to create a clean virtual environment and install requirements.txt (torch, torch-npu, torchvision) from PyPI failed with a 300-second timeout. The root cause: pip was resolving PyTorch 2.11.0 and pulling the entire NVIDIA CUDA toolkit stack including cuda-toolkit (2.4kB meta), cuda-bindings (6.3MB), nvidia-cudnn-cu13 (366MB), nvidia-nccl-cu13 (196MB), nvidia-nvshmem-cu13 (60MB), and more — collectively well over 1GB. This is catastrophic on an Ascend NPU host where none of the NVIDIA CUDA libraries are needed. The retry strategy used `python3 -m venv .venv --system-site-packages` to create a virtual environment that inherits globally installed packages. The host already had torch==2.5.1+cpu, torch-npu==2.5.1, and torchvision==0.20.1+cpu installed at `/usr/local/python3.10.17/lib/python3.10/site-packages/`. This reduced Phase 2 duration from an unbounded network download to under 2 seconds. Verification confirmed `torch.npu.is_available()` returned `True` inside the venv. The execution journal shows Phase 2 took 486 seconds total (including the timeout and retry), whereas with the system-site-packages strategy applied from the start, it would have taken under 2 seconds.

**Key lesson**: Before running `pip install -r requirements.txt` on an Ascend NPU host, always check whether torch-npu is pre-installed system-wide. If `python3 -c "import torch_npu; print(torch_npu.__version__)"` succeeds, use `--system-site-packages` to bypass all network downloads. This pattern applies universally to CUDA-to-NPU migrations conducted on Ascend infrastructure, where the container or VM image includes pre-configured CANN toolchains and torch-npu packages.
